# Review rubric

Use this when the user comes with an existing deck, script, video, or transcript and wants feedback. The goal is **prioritized actionable fixes**, not a full rewrite.

## How to run a review

1. Read everything the user shared (deck text, script, or transcript of their pitch video).
2. Score each line below as **PASS / RISK / FAIL**.
3. Tag which of the 5 `failure-patterns.md` each FAIL or RISK maps to.
4. Return the **top 3–5 highest-leverage fixes only**. Rewriting every slide buries the important stuff.

For each fix, use this format:

> **Slide X — [slide name]** · *Fail: [failure pattern]*
> **Current:** "[exact quote from their draft]"
> **Suggested:** "[rewrite]"
> **Why:** [one sentence]

Short, quotable, actionable. The user reads this in 90 seconds and knows what to change.

## The rubric

### Structural (does the deck follow the framework?)

- [ ] **Intro slide present** (logo + one-liner)
- [ ] **Problem slide present** and sells ONE main problem (not a list)
- [ ] **Problem has data** backing the pain
- [ ] **Value proposition slide** (one sentence, user-facing benefit)
- [ ] **Features / how-it-works slide** before the demo
- [ ] **Demo slide** before the 2:00 mark
- [ ] **Market slide** present (user categories, not boring TAM/SAM/SOM)
- [ ] **Business model slide** present (non-negotiable)
- [ ] **Traction slide** present with 3 real metrics
- [ ] **Growth strategy** present (3 concrete channels)
- [ ] **Roadmap** present (dated, one big thing per quarter)
- [ ] **Team slide** leads with one outstanding achievement
- [ ] **Conclusion + CTA** present (not "thank you")

### Content (is each slide doing its job?)

- [ ] **Hook in first 15 seconds.** Would a tired VC keep watching?
- [ ] **Problem sold.** Can a non-expert judge *feel* the pain after watching the problem slide?
- [ ] **Value prop is benefit-speak**, not feature-speak
- [ ] **Demo is ≤30 seconds** and shows ONE flow (not the whole product)
- [ ] **Demo has no background music**
- [ ] **Traction uses real product metrics** (revenue, active users, transactions, growth rate). **No X/Twitter followers.**
- [ ] **Traction shows a trend** (growing, not a snapshot)
- [ ] **Business model is a specific mechanism** (not "we'll figure out monetization")
- [ ] **Team slide is NOT just bios** — has an outstanding achievement
- [ ] **Closing slide has a specific CTA** (not "thank you" or "contact us")
- [ ] **Something memorable embedded** — one line, stat, or visual the judge would quote to another judge

### Language (tense + word discipline)

- [ ] **No conditionals anywhere outside roadmap.** Grep script for: *trying*, *hope*, *aim*, *would*, *should*, *could*
- [ ] **Present tense for current state, past tense for shipped, future tense ONLY on roadmap slide**
- [ ] **No "obviously" / "of course" / "as you know"**
- [ ] **Slide titles ≤ 4 words**
- [ ] **No slide with more than ~15 words total**

### Delivery (if there's a recorded video or live pitch)

- [ ] **Every slide-to-slide transition is scripted**, not "ok, next slide"
- [ ] **Total runtime ≤ 3:00** (target 2:30–2:45 for breathing room)
- [ ] **Rhythm varies** across beats (tone drops on problem, rises on solution, serious on traction, inviting on close)
- [ ] **Lists use animations** to reveal one item at a time
- [ ] **Founder not reading a script** (second-nature delivery, eye contact with camera/audience)

### Design (if reviewing an actual deck)

- [ ] **Consistent brand identity** (colors, fonts, logo placement)
- [ ] **One slide = one message**
- [ ] **Not Google-Slides-default-template quality.** Even simple is fine — but intentional.
- [ ] **No wall-of-logos partner slide.** Logos have one-line context.

## Severity triage

After scoring, triage fixes by leverage:

**Critical (fix before submission):**
- Missing business model slide
- Missing or hidden CTA
- Conditionals throughout ("trying to", "hope to")
- Demo appears after 2:00
- Team slide = bios only
- X/Twitter followers presented as traction

**High impact (fix if time allows):**
- Weak hook in first 15 seconds
- Problem slide has no data
- Value prop is feature-speak
- Slides over 15 words
- No scripted transitions

**Polish (fix if time really allows):**
- Slide titles 5+ words
- Inconsistent brand identity
- Demo has background music
- No memorable quotable line

In a short review, surface the Critical items first. If the user hasn't hit submission yet and you have bandwidth, surface 1–2 High-impact items too. Don't list Polish items in a 3–5-fix review — they'll distract from the important ones.

---

## What NOT to do in a review

- **Don't rewrite the whole pitch.** The user's voice matters; you're not ghostwriting, you're coaching.
- **Don't compliment-sandwich.** "Great work overall, but…" burns time. Be direct. The founder is on a deadline.
- **Don't list 20 fixes.** 3–5 prioritized fixes get acted on. 20 fixes don't.
- **Don't guess at facts.** If the traction section is missing context, ask the founder for the real numbers before rewriting.
- **Don't apply the framework religiously where it doesn't fit.** If the founder's deck is for a non-Colosseum context (e.g., a technical demo to other devs), some rules bend. Use judgment.
